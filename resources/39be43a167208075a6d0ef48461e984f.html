클로드 데스크톱 앱과 VS코드의 클로드 코드를 구분해서 정리했습니다.

이유 : [VS Code]에서 클로드 코드를 사용하면 스스로 로컬 파일을 읽고 모델을 바꾸며 자율적으로 일하지만, [데스크톱 앱]에서는 사용자가 파일 첨부부터 모델 변경까지 모두 수동으로 제어해야 하기 때문에 프롬프트 일부가 다릅니다.

두 환경의 차이점 정리 (궁금하신 분들만 펼쳐보세요)

1. 모델 전환 (Sonnet ↔ Haiku/Fable)

  • VS Code: *"두 모델을 조합해서 써줘"*라고 던지면, 알아서 모델을 스위칭해가며 똑똑하게 일합니다.
  • Claude Desktop: 사용자가 채팅창에서 직접 Sonnet과 Fable을 바꿔가며 대화를 이어가야 합니다.

2. 파일 및 폴더 접근성

  • VS Code: 채팅창에 파일을 직접 첨부하지 않아도, 내 컴퓨터 속 모든 파일을 스스로 탐색하고 읽어옵니다.
  • Claude Desktop: 내 폴더에 직접 접근할 수 없으므로, 파일 내용을 직접 복사·붙여넣기하거나 클로드 코워크를 활용해 폴더에 접근시켜야합니다

3. 설계도(.md 파일) 연동

  • VS Code: 모델이 이전에 만들어둔 설계도나 가이드라인 문서를 알아서 찾아서 읽고 반영합니다.
  • Claude Desktop: 이전 단계에서 만든 설계도를 다음 대화에 쓰려면, 사용자가 직접 '프로젝트 지침'에 붙여넣거나 파일에 업로드해 두어야 합니다.

4. 작업 연속성 (승인 절차)

  • VS Code: Auto Mode로 켜두면 사용자의 승인 없이 처음부터 끝까지 혼자서 논스톱으로 실행합니다.
  • Claude Desktop: 단계마다 사용자가 코드를 복사하거나 실행을 확인해 주는 수동 승인 과정이 계속 필요합니다.

1. 반복 업무 설계도 제작

언제 쓰나: 매주 반복되는 큰 업무(주간 리포트, 콘텐츠 기획, 리서치)가 있을 때. 실행은 나중에 싼 모델이 해도, 설계는 지금 Fable에게 맡겨요.

7. 하위 모델이 놓치기 쉬운 '암묵 판단 규칙'을 명시적으로 적어라 —
   상위 모델은 알아서 챙기지만 싼 모델은 안 하는 '센스'를 글로 박는 칸이다:
   - 목표의 '의도'를 살리는 법 (예: '인식·이해'류 목표면 개념·비유를 먼저)
   - 사용자가 말 안 한 불안/니즈를 선제적으로 챙길 지점
   - 이 업무에서 반드시 시연·강조해야 할 핵심 포인트

  (그대로 저장해서 다음에 다시 꺼내 쓸 수 있게)
- 마지막에 '테스트 방법'을 붙여: 이 설계도로 실제 작업 1건을 시켜본 뒤,
  결과가 6번(성공 기준)과 7번(암묵 판단 규칙)을 지켰는지 스스로 채점하는
  짧은 체크리스트(5줄 이내).

   - 기준일: 2026-07-01 이후 세션만 후보로 삼는다.
     (내가 Fable 5를 쓰기 시작한 시점. 다르면 내가 정정한다)
   - 검색은 "Fable 5" 같은 이름이 아니라 실제 작업 키워드(한국어)로 하라.
     예: 이메일 초안, 릴스 분석, Notion 위키, Figma, 카드뉴스, 감사
   - 제외: 이 추출 작업 자체를 다룬 메타 세션
   - 각 후보마다 두 가지 근거를 표로 제시하라:
     ① 날짜 (기준일 이후인지)
     ② 관찰된 행동 신호 (예: 실패 투명 보고, 중단 조건 작동,
        fetch 근거만 사용, 자기검증 수행 등 — 기록에서 실제 확인된 것만)
   - 결과가 좋았던 세션 5~10개를 추려라. 부족하면 그 사실을 먼저 보고하라.

   ★ 확인 게이트: 후보 표를 보여주고 멈춰라.
   "이 중 Fable 5가 아닌 세션이 있으면 번호로 빼달라"고 묻고,
   내가 확정한 세션만으로 2단계를 진행하라.

2. [접근 방식 추출 — 추측 금지, '실제로 한 것'만]
   확정된 세션에서 Fable이 실제로 한 행동만 근거로 아래를 뽑아라.
   항목마다 '어느 세션(제목+날짜) 어느 대목'인지 근거를 함께 적어라.
   기록에서 확인 안 되는 건 그럴듯하게 채우지 말고 [기록 없음]으로 남겨라.
   - 문제를 처음에 어떻게 쪼개고 계획했는지 (계획 방식)
   - 무엇을, 어떤 순서로 확인·검증했는지 (검증 순서)
   - 애매한 상황에서 어떻게 결정했는지 (추측 대신 확인·근거를 댄 지점)
   - 끝내기 전에 스스로 무엇을 점검했는지 (자기검증)
   - 이 방식 없이 하면 하위 모델이 놓칠 지점 (= 이식해야 하는 이유)

3. [하위 모델용 지침으로 변환]
   위에서 뽑은 걸, 하위 모델이 '그대로 따라 하면 Fable처럼 일하게 되는'
   지침으로 정리해라. 설명·요약이 아니라 시켜서 실행되는 형태로:
   - 행동 지시 (명령형): "~할 때는 먼저 ~하라 / ~은 반드시 확인하라 /
     끝내기 전 ~로 점검하라"
   - 그 아래 짧은 체크리스트 (하위 모델이 스스로 O/X 확인하게)
   - 특정 업무에서만 통하는 건 빼고 어느 업무에나 적용되는 것만.
     (업무 한정이면 "업무별 후보"로 따로 빼라)
