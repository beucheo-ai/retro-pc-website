# 노션 자료 모음집 동기화 — 작업 설계도

> 이 문서는 `notion-sync.mjs`를 처음 설계할 때(2026-07-17) 겪은 시행착오와 판단 근거를 기록한다.
> **자료를 추가/갱신하는 반복 작업은 이 문서만 보고 실행 모델(Sonnet 등)이 그대로 따라 하면 된다.**
> 설계를 다시 고민해야 하는 상황(사이트 톤 변경, 새 블록 타입 대응 등)에서만 상위 모델 판단이 필요하다.

## 1. 이 작업이 존재하는 이유

- 예전 방식: 자료 카드 클릭 → 노션 페이지를 iframe으로 임베드.
- 문제: 모바일 브라우저에서 노션 자체 스크립트(앱 유도/리다이렉트)가 iframe 안에서도 동작해 페이지가 "튕겨" 나가는 현상 발생.
- 해결: 노션 API로 블록을 읽어와 **사이트 자체 스타일의 정적 HTML로 변환**해 `resources/<id>.html`에 저장하고, 모달은 그 정적 HTML을 `fetch()`해서 보여준다. 노션 임베드 완전 제거 → 모바일 이슈 자체가 사라짐.

## 2. 실행 방법 (반복 작업 — 가벼운 모델도 가능)

```bash
NOTION_TOKEN=ntn_xxx node scripts/notion-sync.mjs
```

- 토큰은 **절대 커밋 금지**. 매번 환경변수로만 넘긴다.
- 새 자료 추가 시: 노션에서 해당 페이지 `···` → `연결 추가` → 통합(이름: "깃허브연동") 연결 → `scripts/notion-sync.mjs`의 `PAGES` 배열에 페이지 링크나 ID를 한 줄 추가 → 스크립트 재실행.
- 스크립트가 `index.html`의 `NOTION_RESOURCES_START ~ END` 마커 사이 `RES` 배열을 통째로 재생성한다. **그 사이 블록은 손으로 수정하지 말 것** (다음 실행에서 덮어써짐).
- 실행 후 반드시 로컬 프리뷰로 확인:
  ```bash
  python3 -m http.server 8934
  # 브라우저: http://localhost:8934/index.html → ③ 자료실 → 카드 클릭
  ```

## 3. 설계 판단 기록 (왜 이렇게 만들었나)

| 판단 | 이유 |
|---|---|
| 노션 iframe 임베드 완전 제거, `fetch()` + `innerHTML` 로컬 렌더링 | 모바일 튕김 문제의 근본 원인 제거. 노션 쪽 스크립트 의존성 자체가 없어짐 |
| 토글은 `<details>/<summary>`로 변환 (기본 닫힘) | 처음엔 "전부 펼쳐서" 요청받아 강제 펼침으로 만들었으나, 긴 문서에서 가독성이 나빠져 네이티브 토글로 변경 (2026-07-17 피드백) |
| **heading_1/2/3의 `is_toggleable: true`도 토글로 처리** | 노션엔 "토글 제목" 블록이 따로 있음(일반 heading과 API 필드가 다름). 이걸 놓치면 그 안에 든 하위 콘텐츠 전체가 통째로 사라짐 — 실제로 "클로드 데스크톱 앱 사용자 버전" 토글들이 빠진 버그가 있었음 |
| `richTextToHtml`에서 `annotations.color` 보존, code 블록도 `richTextToPlain` 대신 `richTextToHtml` 사용 | 노션에서 "이 부분만 직접 수정하세요"처럼 색상/배경으로 강조해둔 텍스트가 있음. plain text로만 뽑으면 이 강조가 사라져서 사용자 피드백으로 수정함 |
| `.res-content`는 사이트 기본 픽셀 폰트(Morris) 대신 시스템 산세리프 폰트 사용, `image-rendering:pixelated` 해제 | 사이트 전역 픽셀 폰트가 짧은 UI 텍스트엔 어울리지만 긴 본문에는 가독성이 매우 나쁨. 모달 내부(`.res-content`)에서만 예외 적용 |
| 이미지는 150KB 넘으면 `sips`로 JPEG 78% 압축 + 최대 너비 1400px로 리샘플 | 노션 스크린샷 원본이 1~2MB급이라 그대로 두면 저장소/모바일 로딩에 부담. macOS 내장 `sips`만 사용 (추가 npm 의존성 없음) |
| desc(카드 설명)는 **최상위 depth(0)의 첫 문단**만 사용, 콜아웃/토글 내부 문단은 제외 | 모든 페이지에 붙어있는 홍보 문구 콜아웃("자료 재가공 및 상업적 판매 금지...")이 첫 문단으로 잘못 뽑히는 버그가 있었음. `DESC_SKIP_PATTERNS` 정규식으로 2중 방어 |
| `url` 필드는 노션 API의 `public_url`이 있을 때만 채움, 없으면 "새 탭에서 열기" 버튼 자체를 숨김 | 워크스페이스 내부 링크(`app.notion.com/...`)는 방문자가 못 열기 때문에 공개 링크가 없으면 버튼을 아예 안 보여주는 게 맞음 |
| Notion API 호출마다 약 340ms 대기 + 429 응답 시 재시도 | 레이트리밋(초당 ~3req) 방지. 27개 페이지 × 블록/토글 재귀 호출이 많아서 필요 |

## 4. 알려진 함정

- **URL 슬러그의 페이지 ID ≠ 실제 API 페이지 ID인 경우가 있었다.** 예전 사이트에 박혀 있던 `notion.site/...-39ee43a1...` 링크가 가리키는 페이지는 통합에 연결되지 않았고, 실제로는 같은 제목의 다른 페이지(다른 ID)가 연결되어 있었다 (아마 노션에서 재구성/복제된 것으로 추정). → 항상 `/v1/search`로 통합이 접근 가능한 페이지 목록을 먼저 확인하고, 거기 있는 ID를 쓸 것. 오래된 URL을 맹신하지 말 것.
- 통합을 페이지 하나하나에 연결하지 않아도, 상위 데이터베이스/페이지에 연결하면 하위 페이지 전체가 한 번에 접근 가능해질 수 있다 (이번에 27개가 한 번에 잡힌 이유).

## 5. 완료 기준 (매번 확인)

1. `git diff --stat` — `resources/`, `index.html`의 RES 배열 부분만 바뀌었는지 확인 (범위 밖 수정 없는지)
2. 로컬 프리뷰에서 새로 추가한 카드 클릭 → 내용/이미지/토글/색상 강조 정상 표시
3. 모바일 뷰포트(예: 390×844)에서도 동일하게 확인
4. 토큰 파일이 프로젝트 폴더 안에 없는지 확인 (`git status`에 안 잡히는지)
5. 커밋 → 푸시 → GitHub Pages 빌드 완료까지 확인 후 배포 URL 스크린샷
